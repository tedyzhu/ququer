# 重构与根因治理方法论

> 本文把 ququer 项目 P0–P5 的模块化重构、以及三个根因治理 spec(标题显示 / 系统消息 / 身份识别)中反复验证有效的手法,沉淀为可复用的方法论。
>
> 适用场景:**没有真机/E2E 验证通道、充满历史 hotfix 复制粘贴、不敢改又必须改的遗留代码**。

## 核心约束:本项目的特殊处境

所有方法都从一个硬现实出发:

- **真机调试通道一直不通** —— 任何动态行为只能靠静态推断,无法跑通端到端验证。
- **历史是 176+ 份 hotfix 堆出来的** —— 同一逻辑被复制粘贴到多处,改一处漏一处,反复回归。
- **业务行为零变化是硬约束** —— 重构/收敛不允许引入未经确认的行为变化。

这三条决定了:**测试是唯一的安全网,而且必须是纯静态、零依赖、可在 CI 秒级跑完的测试**。下面的所有手法本质都是围绕"如何在没有真机时安全地改遗留代码"。

## 一、两种工作模式,先分清再动手

| 模式 | 何时用 | 产物 |
| --- | --- | --- |
| **模块化重构** | 把巨型文件按职责拆成模块 | 薄壳 + attach 模块,行为零变化 |
| **根因治理** | 同一逻辑散落复制、反复 hotfix | 抽权威单一真相 + 收敛 + 测试固化 |

判断依据:

- 如果问题是"一个文件太大、难维护" → 模块化重构(见 §2)。
- 如果问题是"同一个判断逻辑在 N 处复制,改一处漏一处" → 根因治理(见 §3)。
- 如果问题是"动态行为错乱、依赖运行时时序" → **大概率不能在无真机时治理**,只能用测试固化现状 + 文档化(见 §4)。

## 二、模块化重构范式(P0–P5 验证)

### 2.1 attach 薄壳模式

把一段函数体从巨型 `Page({...})` 搬到独立模块,通过 `attach(page)` 在 onLoad 时挂回 page 实例:

```js
// modules/xxx.js
function attach(page) {
  page.someMethod = function() { /* 整段搬迁,this 不动 */ };
}
module.exports = { attach };
```

要点:
- **函数体内 `this` 一字不改** —— 运行时 `this === page`,行为零差异。
- 跨模块依赖通过 `page.xxx()` / `page.data` 调用,不引入新状态容器。

### 2.2 关键边界:wxml 绑定的方法不能 attach

小程序 Page 注册时会快照属性,`bindXxx` 绑定的方法必须在注册那一刻就存在;attach 是 onLoad 时挂的,太晚。

**对策**:抽离前先 `grep -E '"(\w+)"' page.wxml` 检查;wxml 绑定的方法(sendMessage / onShow / onMessageTap 等)保留在主文件。

### 2.3 渐进抽离,每刀独立 PR

- 一个模块一个 PR,一次 commit。
- 大方法分阶段抽(identity-resolver 拆了 5 个阶段、6 个 PR)。
- 高风险段(async + 多副作用 + 多 let 重写)**主动放弃**,不硬抽(P3 的 onLoad 阶段 2d)。

成效:chat.js 15500 → 2237 行(-85.6%),20 个模块,零业务变化。

## 三、根因治理范式(三个 spec 验证)

适用于"同一判定散落复制"型根因。**这是本项目最有价值的沉淀**。五步法:

### Step 1. 实证调研确认根因(不靠猜)

```
grep 散落副本 → 精确比对每个副本的差异 → 判断是"完全一致的复制"还是"语义不同的并存"
```

- 标题治理:发现占位符黑名单 5+ 处,且**各不相同**(有的漏「新用户」,有的用子串 includes)。
- 系统消息治理:发现格式判定 11 处,但**完全一致**的复制粘贴。
- 身份识别:发现 creator/visit 键读写散落,但**语义不同**(不是简单复制)→ 判定不适合收敛。

**这一步决定后面能不能做、怎么做。** 副本一致 → 纯提取(零风险);副本不一致 → 收敛会翻转部分边界(需分层处理);语义不同 → 不收敛。

### Step 2. 抽取权威单一真相(纯函数)

把判定抽成 `chat-helpers.js` 的纯函数,作为唯一来源:

```js
function isASideSystemMessage(content) {
  if (!content || typeof content !== 'string') return false;  // 非法入参安全兜底
  return /* 现状各副本 OR 条件的并集 */;
}
```

要点:**纯函数**(只依赖入参,不读 page/wx/全局),便于细粒度测试;语义 = 现状各副本的**并集**(不增不减判定范围,除非明确要更正)。

### Step 3. 测试先行,固化收敛前的现状

**这是零行为变化保证的核心。** 先写回归测试,把收敛**之前**的实际行为固化成基线:

- 测试用例的期望值 = 当前实现的**实际产出**(不是"应该的产出")。
- 跑绿后,这套测试就成了"行为变化的唯一判据"。

关键技巧——**测试替身必须复刻真机行为**:
- title 治理里 fakePage 的 `isPlaceholderNickname` 必须绑定**真权威实现**(真机上 chat.js 恒这么绑),否则会误判哪些收敛点会翻转。

### Step 4. 逐处收敛,每处后重跑

- 一处一改,改完立即 `bash .tools/run_all_tests.sh`。
- **Tier A 用例(行为应不变)变红 = 真实回归** → 停止,上报,**绝不改期望值蒙混过关**。
- **Tier B 用例(已知边界翻转)** → 显式列出"收敛前 → 收敛后"对照,交用户确认是"接受更正"还是"保留旧行为"。

Tier A/B 分层是关键:它把"零变化的硬证据"和"已知有限的边界翻转"清晰隔离,避免一锅粥。

### Step 5. 全量验证 + 文档化

- 全局 grep 确认无残留散落副本。
- 整理 Tier B 行为更正清单写进测试注释(留痕)。
- tasks.md 记录执行结果 + 后续跟进项。

### 三个 spec 的差异对照(范式的灵活应用)

| Spec | 副本性质 | 处理 | Tier B 翻转 |
| --- | --- | --- | --- |
| 标题显示 | 散落且**不一致** | 收敛 + 边界更正 | 有(「新用户」泄漏进标题 → 更正) |
| 系统消息 | **完全一致**复制 | 纯提取 | 无(全 Tier A) |
| 身份识别 | 语义**不同**并存 | **不收敛**,固化 + 文档化 | — |

**方法论的成熟标志不是"什么都收敛",而是知道什么不该收敛。**

## 四、无真机时的红线:发现 bug ≠ 立刻修

身份识别治理中发现一个真实 bug(访问计数存储键读写不一致,证据静默失效)。**没有直接修**,而是:

1. 用测试**固化 bug 现状**(bug-1 现状失效 / bug-2 对照生效 / bug-3 根因断言)。
2. readme 已知技术债**文档化**,标注"待真机验证后修"。

理由:修复会**改变身份判定这种高风险动态行为**,而无真机无法验证修复是否引入新误判。

**红线**:在没有验证能力时,**改静态可判定的逻辑(格式/字符串/纯函数)是安全的;改依赖运行时时序、身份状态机的动态行为是危险的**。后者即使发现 bug 也先固化 + 文档化,等真机。

## 五、测试基础设施(支撑以上一切)

- **纯 Node 静态测试**:`assert`/`assertEqual` + fakePage + attach,零测试框架/PBT 依赖。
- **fakePage 模式**:`setData` 写入 data + 记录调用;`messages[i].xxx` 路径解析;协作方法 spy。
- **fake timers**:替换全局 setInterval/setTimeout,`runAllTimeouts` / `tickAllIntervals` 验证延迟/轮询逻辑,不让真定时器悬挂。
- **可重写 mock**:`cloudResponse` / `cloudShouldFail` 切换 success/fail/success=false 三态;mock `wx.cloud.database` watch 链捕获 onChange/onError。
- **一键 + CI**:`.tools/run_all_tests.sh` 串联全部测试,接入 GitHub Actions,~10s 跑完,每次 PR 自动守护。

成效:测试套件从 0 专测 → 19 个测试 / 883 PASS。

### 为什么不引入属性化测试(PBT)

评估过,本项目不适合:输入空间有限可枚举(状态 × 身份 × 格式分类)、边界已知有限、项目零依赖纯 Node 约定。**表驱动示例 + 精选边界语料比随机 PBT 更精准**,且不破坏约定。

## 六、工程纪律(贯穿始终)

- **每刀独立 PR + 独立 commit**,中文 commit message。
- **改动后必跑 `run_all_tests.sh` 再提交**。
- **失败循环识别**:同一手法失败两次就停下来诊断根因,换思路,不无限打补丁。
- **降级与放弃要显式**:高风险/低 ROI 的项主动放弃并记录理由(P3 的 onLoad 2d、P4 的 chat-debug-tools 边界整理)。
- **诚实区分"已验证"和"未验证"**:无真机时,明确标注哪些是测试固化的事实、哪些是静态推断。

## 七、一句话总结

> 在没有真机的遗留代码里安全地改东西,靠的不是胆子大,而是:**先用测试把现状焊死,再在测试的保护下做最小改动,改完测试说话——测试绿了才算没破坏,测试红了就停手上报。能收敛的收敛,该等真机的就固化文档化等着。**
