# 真机双端测试指南

> 解决本项目最大痛点:**真机调试通道 Timeout、双端流程无法验证**。
> 本指南给出"两台真手机 + 后台实时日志"的可执行流程。

## 一、前提(缺一不可)

1. **两个不同的微信号**,各登一台手机:
   - 手机甲 = 微信号 A = 发送方(创建者)
   - 手机乙 = 微信号 B = 接收方(被邀请)
   - ⚠️ 必须是两个真实不同的号。同号测试会导致 openId 相同 → 身份误判(历史 bug 根源)。
2. **A、B 两个号互为微信好友**(邀请卡片要发到好友会话)。
3. **A、B 都加为体验成员**:mp.weixin.qq.com → 管理 → 成员管理 → 体验成员。
4. **云函数已部署 + 数据库集合就绪**(users / messages / conversations)。

## 二、跑流程(体验版,最稳)

1. 开发者工具 → 上传 → 填版本号 → 上传
2. mp.weixin.qq.com → 版本管理 → 开发版本 → 选为体验版 → 生成体验版二维码
   - 「体验版设置」的"页面路径(选填)"**留空**,走默认 entryPagePath(登录页)
3. **手机甲(A)**:扫体验版码 → 登录 → 创建新聊天
   - 期望:标题显示 A 自己昵称,系统消息"您创建了私密聊天…"
4. 手机甲 `… → 转发`,把邀请卡片发给 **微信号 B**
5. **手机乙(B)**:微信里点开 A 发来的卡片 → 授权登录 → 自动加入
   - 期望:标题"我和[A昵称]（2）",系统消息"加入[A昵称]的聊天"
6. 双向发消息,验证阅后即焚 / 标题 / 系统消息

> 快速迭代用**预览版**(改完代码点预览重扫码,不用每次上传;且默认带 vConsole)。

## 三、看日志(三条通道)

### 1. 实时日志(后台看真机日志,推荐)

代码已接入 `wx.getRealtimeLogManager`(见 `app/utils/realtime-logger.js`),关键节点已埋点:
- **身份判定**(`resolveFinalIdentity` 出口):tag `[身份判定]`,含 finalIsFromInvite / isActualCreator / chatId / openId
- **B 端加入**(`joinChatByInvite`):tag `[B端加入]`,含 enter / cloud-return / cloud-fail 各阶段
- **标题决策**(`updateDynamicTitle`):tag `[标题决策]`,含 title / participantCount / isFromInvite

**查看**:mp.weixin.qq.com → 开发 → 运维中心 → **实时日志**
- 按时间 + 用户 openId 筛选(A、B 两个号 openId 不同,可分别看两端)
- 搜 `[身份判定]` / `[B端加入]` / `[标题决策]` 定位关键节点

### 2. vConsole(手机上看 Console)

预览版默认带;体验版在 `… → 打开调试`。屏幕左下角绿色按钮,可看全部 `console.log`(项目里 `🔥`/`🔗`/`🏷️` 前缀日志)、网络、Storage。

### 3. 云函数日志(后端链路)

云开发控制台 → 云函数 → 选函数 → 日志。排查"加入失败/收不到消息"先看这里。

### 4. 数据库(身份 bug 的硬证据)

云开发控制台 → 数据库 → `conversations`,看这条聊天的 `participants`:谁是 creator/joiner、有无 temp_user 污染、是否重复。

## 四、每轮清缓存(测身份 bug 必做)

身份误判与本地残留存储强相关。每测一个场景前,两台手机都清:
- 微信 → 我 → 设置 → 通用 → 存储空间 → 小程序缓存,清本小程序
- 清完把小程序从"最近使用"删掉重进(iOS 缓存顽固)

## 五、重点验证清单

| 场景 | 期望 | 验证的 bug |
| --- | --- | --- |
| A 创建(B 未加入) | 标题=A 昵称;"您创建了私密聊天…" | 创建态身份 |
| B 加入 | B 端标题"我和[A]（2）";"加入[A]的聊天" | 邀请链路 / B 端身份 |
| A 端(B 加入后) | 标题"我和[B]（2）";"[B]加入聊天" | A 端参与者监听 |
| **A 退出后同号回访同一聊天** | **标题仍"我和[B]（2）",不误判成 B 端** | **#42 访问计数 bug** |

> 最后一行是验证 #42 那个已知 bug(`resolveFinalIdentity` 访问计数读错存储键导致频繁访问证据失效)的关键场景。配合实时日志 `[身份判定]` 看 `isActualCreator` 是否如期为 true。

## 六、常见坑

- 两个号是同一个 → openId 相同,身份必混淆(头号坑)
- 没加体验成员 → B 扫码打不开(权限问题,非 bug)
- 云函数没部署 / 环境 ID 不对 → 加入、收发全失败
- 卡片发给"文件传输助手" → 只到自己,要发到 A↔B 好友会话
